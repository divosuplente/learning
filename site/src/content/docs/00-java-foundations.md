## 1. Java 21 Records

Records are Java's answer to the boilerplate problem. Before records, a simple data class required a constructor, getters, `equals`, `hashCode`, and `toString` — easily 50+ lines for something that should be one line.

A record is an **immutable data carrier**. The compiler generates the constructor, getters, `equals`, `hashCode`, and `toString` for you.

```java
import java.math.BigDecimal;

public record Customer(
        String id,
        String name,
        String email,
        BigDecimal balance) {}
```

That's it. You get `customer.id()`, `customer.name()`, `customer.equals(other)`, and a readable `toString()` — all generated by the compiler.

### Compact Constructor with Validation

Sometimes you need to validate fields during construction. Records support a **compact constructor** — a constructor body without parameters:

```java
public record Order(
        String id,
        Product product,
        Customer buyer,
        BigDecimal total,
        java.time.LocalDateTime timestamp) {

    // Compact constructor — runs before field assignment
    public Order {
        if (total.compareTo(BigDecimal.ZERO) < 0) {
            throw new IllegalArgumentException("Order total cannot be negative");
        }
        if (buyer == null || product == null) {
            throw new NullPointerException("Buyer and product are required");
        }
    }
}
```

### Records Can Nest Other Records

Records compose naturally — one record can contain another:

```java
public record Product(String sku, String name, int stock, BigDecimal price) {}
public record OrderItem(Product product, int quantity, BigDecimal lineTotal) {}
public record Order(String id, Customer buyer, List<OrderItem> items, BigDecimal grandTotal) {}
```

### Why Records Matter

- **Immutable by default** — no setters, no defensive copies, thread-safe out of the box.
- **Less boilerplate** — one line instead of 50. The intent is clear.
- **Pattern matching works naturally** — record destructuring in `switch` (see section 3).
- **Perfect for DTOs** — data transfer objects that just carry data between layers.
- **`equals` and `hashCode` are correct** — the compiler generates them based on all fields, so you won't forget a field.

> **Note**: Records are **reference types**, not primitives. They cannot be annotated with `@Entity` (for JPA) directly — you'll need a wrapper or use Spring Data's record support (covered in Module 04).

## 2. Generics

Generics let you write **type-safe, reusable code**. Without generics, you'd use `Object` everywhere and cast at runtime — which means bugs only surface when the code runs, not when it compiles.

```java
import java.util.List;
import java.util.ArrayList;
import java.util.function.Function;

// A generic container that holds items of any type T
class Box<T> {
    private final List<T> items = new ArrayList<>();

    public void add(T item) {
        items.add(item);
    }

    public T get(int index) {
        return items.get(index);
    }

    public int size() {
        return items.size();
    }
}
```

Now `Box<String>` only accepts strings, and `Box<Integer>` only accepts integers. The compiler catches type mismatches before runtime:

```java
Box<String> stringBox = new Box<>();
stringBox.add("hello");  // ✅ fine
stringBox.add(42);        // ❌ compile-time error

Box<Integer> intBox = new Box<>();
intBox.add("oops");      // ❌ compile-time error
```

### Bounded Type Parameters

Sometimes you need to constrain `T` to a specific hierarchy. Use `extends` to set an upper bound:

```java
// T must be a Number or a subtype of Number
public class Statistics<T extends Number> {
    private final List<T> data = new ArrayList<>();

    public void add(T value) {
        data.add(value);
    }

    public double average() {
        if (data.isEmpty()) return 0;
        double sum = 0;
        for (T n : data) {
            sum += n.doubleValue();  // Number.doubleValue() is available because T extends Number
        }
        return sum / data.size();
    }
}
```

This lets you call `n.doubleValue()` because `T` is guaranteed to be a `Number`. Without the bound, the compiler wouldn't know `T` has that method.

### Wildcards: `? extends T` and `? super T`

Wildcards make generic APIs more flexible. There are three forms:

```java
// PECS: Producer Extends, Consumer Super

// ? extends T — reading from a collection of T or subtypes
public double sumAll(List<? extends Number> numbers) {
    double sum = 0;
    for (Number n : numbers) {  // safe to read
        sum += n.doubleValue();
    }
    return sum;
}

// ? super T — writing to a collection of T or supertypes
public void addNumbers(List<? super Integer> list) {
    for (int i = 1; i <= 10; i++) {
        list.add(i);  // safe to write Integer
    }
}

// ? — unbounded wildcard, only reading Object
public void printSize(List<?> list) {
    System.out.println(list.size());  // can't add anything, but size() works
}
```

**PECS rule**: If you **read** from a collection, use `? extends T`. If you **write** to it, use `? super T`. If you do both, don't use a wildcard.

### Type Erasure

Generics exist only at compile time. At runtime, `Box<String>` and `Box<Integer>` are both just `Box`. This is called **type erasure**:

```java
Box<String> a = new Box<>();
Box<Integer> b = new Box<>();

// At runtime, both are the same class
System.out.println(a.getClass() == b.getClass());  // true — both are Box.class
```

This means you **cannot** do `new T()`, `T.class`, or `instanceof T` at runtime. If you need the class at runtime, pass it explicitly:

```java
public <T> List<T> filterByType(List<?> items, Class<T> type) {
    List<T> result = new ArrayList<>();
    for (Object item : items) {
        if (type.isInstance(item)) {
            result.add(type.cast(item));
        }
    }
    return result;
}
```

## 3. Pattern Matching

Java 21 introduced **pattern matching for `switch`** — you can match on types, record components, and add guard conditions. This replaces long `if-else` chains with expressive, exhaustive switch statements.

```java
record Point(double x, double y) {}
record Circle(Point center, double radius) {}
record Rectangle(Point topLeft, double width, double height) {}

public String describeShape(Object shape) {
    return switch (shape) {
        case null -> "No shape";
        case Circle(Point center, double radius) when radius == 0 ->
            "A point at " + center;
        case Circle(Point center, double radius) ->
            "Circle at " + center + " with radius " + radius;
        case Rectangle(Point topLeft, double w, double h) ->
            "Rectangle at " + topLeft + " (" + w + "x" + h + ")";
        default -> "Unknown shape: " + shape;
    };
}
```

### How It Works

- **Type patterns** (`case Circle c`) match the runtime type and bind a variable.
- **Record patterns** (`case Circle(Point center, double radius)`) destructure records, extracting components directly.
- **Guard clauses** (`when radius == 0`) add extra conditions to a pattern.
- **Exhaustiveness** — if you're matching on a sealed type (see below), the compiler checks you've covered all cases.

### Pattern Matching with Records

Records work seamlessly with pattern matching because their components are accessible:

```java
public sealed interface Payment permits CashPayment, CardPayment, BankTransfer {}

public record CashPayment(BigDecimal amount, String currency) implements Payment {}
public record CardPayment(BigDecimal amount, String cardLast4) implements Payment {}
public record BankTransfer(BigDecimal amount, String iban) implements Payment {}

public String describePayment(Payment payment) {
    return switch (payment) {
        case CashPayment(BigDecimal amount, String currency) ->
            "Cash: " + amount + " " + currency;
        case CardPayment(BigDecimal amount, String cardLast4) ->
            "Card: " + amount + " (****" + cardLast4 + ")";
        case BankTransfer(BigDecimal amount, String iban) ->
            "Transfer: " + amount + " to " + iban;
    };
}
```

No `default` needed — the switch is exhaustive because `Payment` is sealed and all permitted subtypes are covered.

## 4. The `var` Keyword

`var` lets the compiler infer the type of a local variable. It doesn't change runtime behavior — it's purely a compile-time convenience.

```java
// Without var
Map<String, List<Customer>> customersByCity = new HashMap<>();

// With var — same type, less visual noise
var customersByCity = new HashMap<String, List<Customer>>();
```

### When `var` Helps

- **Long generic types** where the type is obvious from the initializer.
- **`var` in `try-with-resources`** — `try (var conn = ds.getConnection()) { ... }`
- **`var` in `for` loops** — `for (var entry : map.entrySet()) { ... }`

```java
// var shines when the type is obvious from the right side
var name = "Alice";                  // String
var price = new BigDecimal("19.99"); // BigDecimal
var orders = List.of(order1, order2); // List<Order>
var stream = orders.stream()          // Stream<Order>
    .filter(o -> o.total().compareTo(BigDecimal.valueOf(100)) > 0);
```

### When `var` Hurts

- **Unclear types** — `var result = process(data)` tells you nothing about the type.
- **Interface vs implementation** — `var list = new ArrayList<String>()` infers `ArrayList`, not `List`. You lose the interface abstraction.
- **Primitive type surprises** — `var x = 0` is `int`, `var y = 0.0` is `double`, `var z = 0L` is `long`. These are all different.

> **Rule of thumb**: Use `var` when the type is obvious from the initializer (constructor call, factory method). Don't use it when the type requires reading documentation or the method name to understand.

## 5. Switch Expressions

Java 14 introduced **switch expressions** — switch as an expression that returns a value, not just a statement. Combined with Java 21's pattern matching, this replaces the old fall-through `switch` with a clean, expressive form.

```java
// Old style — verbose, fall-through, error-prone
String dayType;
switch (day) {
    case MONDAY:
    case TUESDAY:
    case WEDNESDAY:
    case THURSDAY:
    case FRIDAY:
        dayType = "Weekday";
        break;
    case SATURDAY:
    case SUNDAY:
        dayType = "Weekend";
        break;
    default:
        dayType = "Unknown";
}

// New style — expression, no fall-through, returns a value
String dayType = switch (day) {
    case MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY -> "Weekday";
    case SATURDAY, SUNDAY -> "Weekend";
    default -> "Unknown";
};
```

### Arrow Labels vs Colon Labels

- **Arrow labels** (`->`) — no fall-through, single expression or block.
- **Colon labels** (`:`) — fall-through (you must `break`), the old behavior.

You can mix them, but you shouldn't. Stick with arrow labels for clarity.

### Block Bodies in Switch Expressions

If a case needs multiple statements, use a block with `yield`:

```java
String discount = switch (customer.tier()) {
    case BRONZE -> "5%";
    case SILVER -> "10%";
    case GOLD -> {
        if (customer.years() > 5) {
            yield "25% (loyal gold)";
        }
        yield "15%";
    }
    case PLATINUM -> "30%";
};
```

### Pattern Matching in Switch

Java 21 combines switch expressions with pattern matching (see section 3 for the full example):

```java
String result = switch (order.status()) {
    case PENDING -> "Order received";
    case CONFIRMED -> "Processing started";
    case SHIPPED -> {
        var trackingUrl = "https://track.example.com/" + order.id();
        yield "Shipped — track at " + trackingUrl;
    }
    case DELIVERED -> "Delivered";
    case CANCELLED -> "Cancelled";
};
```

## 6. Collections Framework

The Java Collections Framework (JCF) provides the essential data structures: `List`, `Set`, `Map`, and `Queue`. These are the building blocks for almost everything you'll do in Spring Boot.

### List — Ordered, Allows Duplicates

```java
List<String> names = new ArrayList<>();
names.add("Alice");
names.add("Bob");
names.add("Alice");  // allowed — duplicates OK
System.out.println(names);  // [Alice, Bob, Alice]
System.out.println(names.get(0));  // Alice

// Immutable list (cannot add/remove)
List<Integer> numbers = List.of(1, 2, 3, 4, 5);
```

### Set — Unique Elements Only

```java
Set<String> unique = new HashSet<>();
unique.add("Alice");
unique.add("Bob");
unique.add("Alice");  // ignored — already present
System.out.println(unique.size());  // 2

// Immutable set
Set<String> colors = Set.of("red", "green", "blue");
```

### Map — Key-Value Pairs

```java
Map<String, BigDecimal> pricing = new HashMap<>();
pricing.put("premium-widget", new BigDecimal("29.99"));
pricing.put("basic-widget", new BigDecimal("9.99"));
pricing.put("premium-widget", new BigDecimal("34.99"));  // overwrites previous

// Iterate over entries
pricing.forEach((sku, price) -> System.out.println(sku + ": " + price));

// Get with default
BigDecimal price = pricing.getOrDefault("unknown", BigDecimal.ZERO);
```

### Queue and Deque

```java
// Queue — FIFO (first in, first out)
Queue<String> taskQueue = new LinkedList<>();
taskQueue.add("task-1");
taskQueue.add("task-2");
String next = taskQueue.poll();  // "task-1" (removed)

// Deque — double-ended queue, can be used as stack
Deque<String> stack = new ArrayDeque<>();
stack.push("bottom");
stack.push("top");
String top = stack.pop();  // "top"
```

### Streams — Functional Data Processing

Streams let you process collections declaratively — describe **what** you want, not **how** to iterate:

```java
List<Order> orders = List.of(
    new Order("O001", new BigDecimal("150.00"), OrderStatus.CONFIRMED),
    new Order("O002", new BigDecimal("50.00"), OrderStatus.PENDING),
    new Order("O003", new BigDecimal("300.00"), OrderStatus.CONFIRMED),
    new Order("O004", new BigDecimal("75.00"), OrderStatus.CANCELLED)
);

// Filter confirmed orders, extract totals, sum them
BigDecimal totalConfirmed = orders.stream()
    .filter(o -> o.status() == OrderStatus.CONFIRMED)
    .map(Order::total)
    .reduce(BigDecimal.ZERO, BigDecimal::add);
// 450.00

// Group by status
Map<OrderStatus, List<Order>> byStatus = orders.stream()
    .collect(Collectors.groupingBy(Order::status));
// {CONFIRMED=[O001, O003], PENDING=[O002], CANCELLED=[O004]}
```

**Key stream concepts:**
- **Intermediate operations** (`filter`, `map`, `sorted`) — lazy, return a new stream.
- **Terminal operations** (`collect`, `reduce`, `forEach`, `count`) — trigger processing, produce a result.
- **Short-circuiting** (`findFirst`, `anyMatch`, `limit`) — stop once the result is known.

> **Tip**: Keep streams as processing pipelines — don't collect to a list just to iterate it again. Chain operations directly.

## 7. Exception Handling

Exceptions signal that something went wrong. Java distinguishes **checked** exceptions (the compiler forces you to handle them) from **unchecked** exceptions (runtime errors you may or may not catch).

### Checked vs Unchecked

```java
import java.io.IOException;
import java.sql.SQLException;

// Checked exception — compiler forces you to declare or catch it
public String readFile(String path) throws IOException {
    // IOException is checked — you MUST handle it
    throw new IOException("File not found: " + path);
}

// Unchecked exception — compiler doesn't force handling
public int divide(int a, int b) {
    if (b == 0) {
        throw new IllegalArgumentException("Cannot divide by zero");  // RuntimeException subclass
    }
    return a / b;
}
```

**Checked exceptions** are for conditions the caller should reasonably handle — file I/O errors, network failures, database issues. **Unchecked exceptions** are for programming errors — null pointers, illegal arguments, index out of bounds.

### Try-With-Resources

Java 7+ automatically closes resources that implement `AutoCloseable`. No more `finally` blocks for cleanup:

```java
// try-with-resources — conn is auto-closed, even if an exception is thrown
try (Connection conn = DriverManager.getConnection(url, user, password)) {
    var stmt = conn.createStatement();
    var rs = stmt.executeQuery("SELECT * FROM customers");
    while (rs.next()) {
        System.out.println(rs.getString("name"));
    }
} catch (SQLException e) {
    throw new RuntimeException("Database query failed", e);
}
// conn.close() is called automatically here
```

You can declare multiple resources — they close in reverse order:

```java
try (var conn = ds.getConnection();
     var stmt = conn.prepareStatement("SELECT * FROM orders WHERE id = ?");
     var rs = stmt.executeQuery()) {
    // process results
} catch (SQLException e) {
    log.error("Query failed", e);
}
```

### Custom Exceptions

Define your own exceptions to make error handling domain-specific:

```java
public class OrderException extends RuntimeException {
    public OrderException(String message) {
        super(message);
    }
    public OrderException(String message, Throwable cause) {
        super(message, cause);
    }
}

public class OrderNotFoundException extends OrderException {
    public OrderNotFoundException(String orderId) {
        super("Order not found: " + orderId);
    }
}
```

Use unchecked exceptions for most domain errors — it's cleaner than propagating `throws` through every method. Spring Boot itself favors unchecked exceptions.

### Best Practices

- **Catch specific exceptions**, not `catch (Exception e)` — you'll mask real problems.
- **Never swallow exceptions** — at minimum, log them. Empty catch blocks are bugs waiting to happen.
- **Wrap and rethrow** — if you catch a low-level exception (SQL, IO), wrap it in a domain exception with context.
- **Keep try blocks small** — only wrap the code that might throw, not entire methods.

## 8. Packages and Imports

Packages are Java's way of organizing code into namespaces. They prevent naming conflicts and control visibility.

```java
package com.example.ordermgmt.domain;

import java.math.BigDecimal;
import java.time.LocalDateTime;
import java.util.List;
import java.util.Map;
// Static import — use constants/methods without class prefix
import static java.util.Collections.emptyList;

public record Order(
        String id,
        Customer buyer,
        List<OrderItem> items,
        BigDecimal total,
        LocalDateTime createdAt) {

    public List<OrderItem> items() {
        return items != null ? items : emptyList();
    }
}
```

### Visibility Modifiers

| Modifier | Same class | Same package | Subclass (any package) | Anywhere |
| --- | --- | --- | --- | --- |
| `public` | Yes | Yes | Yes | Yes |
| `protected` | Yes | Yes | Yes | No |
| *package-private* (default) | Yes | Yes | No | No |
| `private` | Yes | No | No | No |

Use the most restrictive visibility that works:
- **`private`** for fields and internal helpers.
- **Package-private** for classes only used within the same package.
- **`public`** for API surface (domain records, service interfaces, controller methods).

### Package Conventions

```
com.example.ordermgmt
├── domain/         # Records, enums, value objects — no dependencies on other layers
├── repository/     # Data access interfaces (Spring Data JPA in Module 04)
├── service/        # Business logic — depends on domain and repository
├── controller/     # HTTP endpoints — depends on service
├── dto/            # Request/response objects for the API layer
└── config/         # Spring Boot configuration classes
```

This structure enforces a one-way dependency rule: `controller → service → repository → domain`. No layer depends on a layer above it.

## 9. Object-Oriented Programming (OOP)

Java is fundamentally object-oriented. Even with records and pattern matching, you'll work with classes, interfaces, and inheritance every day.

### Classes

A class defines fields (state) and methods (behavior):

```java
public class BankAccount {
    private final String accountId;
    private BigDecimal balance;

    public BankAccount(String accountId, BigDecimal initialBalance) {
        this.accountId = accountId;
        this.balance = initialBalance;
    }

    public void deposit(BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            throw new IllegalArgumentException("Deposit must be positive");
        }
        balance = balance.add(amount);
    }

    public void withdraw(BigDecimal amount) {
        if (amount.compareTo(balance) > 0) {
            throw new IllegalStateException("Insufficient funds");
        }
        balance = balance.subtract(amount);
    }

    public BigDecimal getBalance() {
        return balance;
    }

    public String getAccountId() {
        return accountId;
    }
}
```

### Interfaces

An interface defines a contract — what methods a class must implement, without specifying how:

```java
public interface PriceCalculator {
    BigDecimal calculatePrice(Order order);
}

public class StandardPriceCalculator implements PriceCalculator {
    @Override
    public BigDecimal calculatePrice(Order order) {
        return order.items().stream()
            .map(OrderItem::lineTotal)
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    }
}

public class DiscountPriceCalculator implements PriceCalculator {
    private final BigDecimal discountRate;

    public DiscountPriceCalculator(BigDecimal discountRate) {
        this.discountRate = discountRate;
    }

    @Override
    public BigDecimal calculatePrice(Order order) {
        var subtotal = order.items().stream()
            .map(OrderItem::lineTotal)
            .reduce(BigDecimal.ZERO, BigDecimal::add);
        return subtotal.multiply(BigDecimal.ONE.subtract(discountRate));
    }
}
```

This is the foundation of **polymorphism** — code can depend on `PriceCalculator` without knowing which implementation it gets. We'll see how Spring Boot uses this with dependency injection in Module 02.

### Abstract Classes

Abstract classes are partially implemented — they can have both abstract methods (no body) and concrete methods:

```java
public abstract class BaseEntity {
    protected String id;
    protected java.time.LocalDateTime createdAt;

    public BaseEntity(String id) {
        this.id = id;
        this.createdAt = java.time.LocalDateTime.now();
    }

    // Concrete method — subclasses inherit this
    public String getId() {
        return id;
    }

    // Abstract method — subclasses must implement
    public abstract boolean isValid();
}

public class Customer extends BaseEntity {
    private final String email;

    public Customer(String id, String email) {
        super(id);
        this.email = email;
    }

    @Override
    public boolean isValid() {
        return id != null && !id.isBlank() && email != null && email.contains("@");
    }
}
```

**Abstract class vs interface**: Use abstract classes when there's shared state (fields) and partial implementation. Use interfaces when you only need a contract. Java interfaces can have default methods (since Java 8), so the line is blurrier — prefer interfaces for flexibility (a class can implement multiple interfaces but extend only one class).

### Enums

Enums are type-safe constants. Each constant is a singleton instance of the enum class:

```java
public enum OrderStatus {
    PENDING("Order placed, waiting for confirmation"),
    CONFIRMED("Payment received, preparing shipment"),
    SHIPPED("Package sent, in transit"),
    DELIVERED("Package delivered to customer"),
    CANCELLED("Order cancelled");

    private final String description;

    OrderStatus(String description) {
        this.description = description;
    }

    public String getDescription() {
        return description;
    }
}
```

Enums can have fields, methods, and even implement interfaces:

```java
// Enum with behavior
public enum ShippingMethod {
    STANDARD {
        @Override
        public BigDecimal cost(BigDecimal weight) {
            return weight.multiply(new BigDecimal("5.00"));
        }
    },
    EXPRESS {
        @Override
        public BigDecimal cost(BigDecimal weight) {
            // Base rate + per-kg surcharge
            return new BigDecimal("15.00").add(weight.multiply(new BigDecimal("2.50")));
        }
    },
    SAME_DAY {
        @Override
        public BigDecimal cost(BigDecimal weight) {
            return new BigDecimal("50.00");  // flat rate
        }
    };

    public abstract BigDecimal cost(BigDecimal weight);
}
```

### Sealed Classes and Interfaces

Java 17 introduced **sealed** types — you control which classes can extend or implement:

```java
public sealed interface Shape
    permits Circle, Rectangle, Triangle {}

public record Circle(double radius) implements Shape {}
public record Rectangle(double width, double height) implements Shape {}
public record Triangle(double base, double height) implements Shape {}

// No other class can implement Shape — the compiler forbids it.
// This enables exhaustive pattern matching (see section 3).
```

Sealed types are powerful for domain modeling — they encode "these are the only possible variants" directly in the type system. Combined with pattern matching, the compiler verifies you've handled every case.

### The Four Pillars

1. **Encapsulation** — hide internal state behind methods. Use `private` fields and expose only what's needed. Records do this automatically.
2. **Inheritance** — `extends` for classes, `implements` for interfaces. A subclass gets the parent's fields and methods.
3. **Polymorphism** — one interface, many implementations. `PriceCalculator` can be standard or discounted — the caller doesn't care which.
4. **Abstraction** — define what a type does (interface) separately from how it does it (implementation). Dependencies point at abstractions, not concrete classes.
